## 📌 Brief Summary
에이전트가 SKILL.md를 grep 부분 검색만 하고 전문을 읽지 않아, 모든 규칙(황금 시간대, 분 jitter, 제목 규칙, 장르 매핑)을 빼먹은 사건. 발행 시간을 3번이나 수정하는 사태 발생. 대표님 직접 지시로 "SKILL.md 전문 읽기"를 최상위 절대 규칙으로 등록.

## 📖 Core Content

### 사건 (2026-04-24)
1. 영상 3개 제작 완료 후 발행 스케줄 설정 시작
2. 에이전트가 SKILL.md에 명시된 슬롯 스케줄(KST 21:00/01:00/06:00)을 무시하고 임의 시간 제안
3. 대표님이 "skill에 황금시간대 있잖아" 지적 → grep 검색으로 찾아서 수정
4. 분 jitter(봇 탐지 방지)도 빼먹음 → 대표님 재지적
5. 3슬롯 배치 대신 3일×Slot2 로 잘못 업로드 → 예약 시간 재수정
6. 총 3번 수정, 대표님 시간 10분+ 낭비

### 근본 원인
- SKILL.md를 세션 시작 시 전문 읽기하지 않음
- 필요할 때 grep으로 부분 검색만 함
- 검색 결과만 보고 전체 맥락(다른 규칙과의 연관)을 놓침

### 절대 규칙 (SKILL.md Section 0 최상단에 등록)
1. 세션 시작 시 SKILL.md를 view_file로 1행~마지막행 전문 읽기 (800줄 이하이므로 1회 호출)
2. 작업 전 체크리스트 자기검증 (슬롯/jitter/제목/장르/음악이미지독립/WAV변환/Python분리)
3. grep 부분 검색만으로 규칙 확인 완료로 판단하는 것 금지
4. KNOWLEDGE_BASE.md, prompt_templates.json도 함께 읽기

### 교훈
- "읽었다"와 "적용했다"는 다르다
- grep 검색은 **보조 수단**일 뿐, 전문 읽기를 대체할 수 없다
- 규칙은 독립적이 아니라 서로 연결되어 있으므로 전체를 알아야 개별 규칙을 올바르게 적용할 수 있다

최근 영상 3개의 퍼포먼스를 YouTube Data API v3로 분석해 "정지 이미지 1장 슬라이드쇼" 구조적 한계를 확인하고, Render Engine v6(멀티씬 비트싱크 컷)을 구축하여 AI 이미지 vs 실사 영상 2가지 프로토타입을 A/B 테스트로 예약 발행했다.

## 📖 Core Content

### 1. 최근 영상 3개 퍼포먼스 분석 (YouTube Data API v3)

| 영상 | 조회수 | 좋아요 | 댓글 | 문제점 |
|------|--------|--------|------|--------|
| Coffee & Late Night Code ☕ | ~94 | 0 | 1 (자동) | 리텐션 낮음, 비주얼 반복 |
| Headphones On, World Off 🎧 | ~87 | 0 | 1 (자동) | 동일 이미지 구조, 차별화 실패 |
| Watch Ticking at 2 AM ⌚ | ~112 | 1 | 1 (자동) | 제목만 바꾸고 영상은 동일 |

- **핵심 발견:** 댓글 1개는 전부 API로 자동 생성한 Pin 댓글 → 진짜 시청자 참여 **제로**
- **평균 조회수 ~94뷰:** 60개 영상에 구독자 4명 = 알고리즘 추천 자체가 안 되고 있음
- **리텐션 패턴:** 제목에 물리적 오브젝트(Coffee/Headphones/Watch) 활용은 CTR에 미미한 효과
- **근본 원인:** 비주얼이 전부 동일 → 시청자가 "같은 영상" 인식 → 리텐션 하락 → 알고리즘 비추천 악순환

### 2. 문제 진단 — 냉정한 자기평가
- v4 엔진 = 이미지 1장 + Ken Burns → **모든 영상이 동일하게 보이는 치명적 문제**
- Pin 댓글 = API 자동 생성 → "시청자 참여"라고 착각한 오류 인정
- Chill 장르 전환, 추상적 제목 등 메타데이터 최적화는 비주얼 문제를 해결 못함
- **결론: 메타데이터 튜닝이 아니라 콘텐츠 비주얼 자체를 바꿔야 한다**

### 3. Render Engine v6 — 핵심 기술 변경
- 1장 → **5장 씬 로테이션** (4비트마다 전환)
- 트랜지션: Flash / Glitch / Zoom-Punch (비트 동기화)
- 가사 스타일 텍스트 오버레이 (비트마다 1줄씩)
- 네온 그라데이션 비주얼라이저 (시안→마젠타)
- 루프 구조 (마지막→첫 씬 블렌드)
- 테마: Cyberpunk City (프로토타입), 추후 3종 로테이션 예정
- PIL 호환성: `Image.ANTIALIAS` → `Image.Resampling.LANCZOS` 패치 필수

### 4. Phase 1: Pexels 무료 실사 영상 ✅ 성공
- Pexels CDN에서 Python requests로 직접 다운로드
- HD(1080p) 접근 시 403 → SD(360p) fallback 전략으로 해결

# 🧠 MrBeast 유튜브 전략: 알고리즘 최적화 프레임워크

## 🎯 개요 (Overview)
이 문서는 'MrBeast 유튜브 전략' 지식을 **데이터 기반의 실행 가능한 프레임워크**로 구조화한 것입니다. 목표는 단순한 콘텐츠 제작을 넘어, 시청 지속 시간과 참여율을 극대화하여 알고리즘 방어율(Algorithm Defense Rate)을 최적화하는 데 있습니다.

## 📌 핵심 원칙 (Core Principles)
1.  **데이터 우선주의:** 모든 전략은 실제 영상의 KPI(조회수, 좋아요, 댓글)와 풀 스크립트 분석을 통해 도출되어야 한다.
2.  **극단적 후킹 (Extreme Hooking):** 시청자가 즉시 멈추고 다음 내용을 보게 만드는 충격적인 질문이나 상황 제시가 필수다.
3.  **긴장 유지 (Sustained Tension):** 목표 달성 과정에서 예상치 못한 장애물과 극적인 전환을 삽입하여 시청자의 이탈을 방지한다.

## ⚙️ 3단계 실행 프레임워크 (The 3-Step Execution Framework)

| 단계 | 목표 | 주요 행동 지침 (Agent Directive) | 알고리즘 영향 |
| :--- | :--- | :--- | :--- |
| **1. 문제 정의** | 시청자의 즉각적인 호기심/욕구를 극대화한다. | 가장 충격적이거나 도전적인 상황을 첫 5~10초에 제시한다. (Hooking) | 초기 클릭률(CTR) 증대 |
| **2. 프레임워크 적용** | 시청자가 끝까지 몰입하게 만든다. | 목표 달성을 위한 과정에서 예상치 못한 장애물과 극적인 전환을 삽입하여 긴장감을 유지한다. (Tension Build-up) | 시청 지속 시간(Watch Time) 증대 |
| **3. 최종 해결책** | 기대치를 충족시키고 강력한 만족감을 제공한다. | 클라이맥스에서 제시된 결과가 모든 긴장감을 해소하거나, 예상치 못한 새로운 차원의 보상을 제공해야 한다. (Resolution & Climax) | 참여율(Engagement Rate) 증대 |

## 📊 알고리즘 방어율 측정 지표
*   **KPI 분석:** 조회수 대비 시청 지속 시간 비율을 핵심 지표로 설정한다.
*   **스크립트 분석:** 대본 내에서 이탈 가능성이 높은 지점(Pain Points)을 식별하고, 이를 회피하는 방식으로 서사를 구성한다.
