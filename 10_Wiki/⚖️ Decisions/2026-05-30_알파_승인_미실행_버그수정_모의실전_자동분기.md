### 세부 내용
- **에러 격리 패턴** — 순차 실행되는 독립 작업들은 반드시 각각의 try-catch로 감싸야 함. 하나의 에러가 후속 작업을 차단하는 것을 방지
- **URL 기반 환경 분기** — 하드코딩된 ID를 환경변수(URL) 기반 조건 분기로 교체하면, 환경변수 변경만으로 모드 전환 가능
- **KIS API 토큰 특성** — 24시간 만료, 6시간 재발급 제한, `EGW00201` 에러로 토큰 만료 식별
- **Cloudflare Worker cron** — `*/5 13-21 * * 1-5` (UTC) = KST 22:00~06:00 매 5분, `scheduled()` 핸들러로 처리
- **KV 기반 예약 시스템** — `pending_approval`, `pending_sell`, `pending_fill_check` 3종 KV로 예약/추적 관리
- `/debug` 엔드포인트로 KV 상태 조회 → `pending_approval` 잔존 확인
- KIS API 응답에서 `EGW00201` (토큰 만료) 에러 확인
- `scheduled()` 핸들러 코드 추적 → 단일 try-catch 구조 발견
- 기존 단일 try-catch → A-0, A, B 3개 독립 try-catch로 분리
- B 섹션의 catch에서는 `sendMessage()`로 에러 알림 발송 (사용자 인지 가능)
- A-0, A 섹션의 catch는 console.log만 (내부에서 자체 에러 처리 로직 존재)
- 코드 수정: **0개** (자동 분기 적용 완료)
- 두 시스템은 독립적이라 각각 변경 필요 (GitHub Secrets ↔ Cloudflare 연동 불가)
- **"하나의 try-catch가 깔끔하다" 오해:** 순차적이지만 독립적인 작업들을 하나의 try-catch에 넣으면, 선행 작업의 에러가 후속 작업을 차단함 → 각 작업을 독립 try-catch로 격리해야 함
- **"토큰 에러는 일시적이다" 오해:** KIS 토큰은 24시간 만료, 6시간 재발급 제한이 있어서 특정 시간대에 반복 실패 가능 → 토큰 에러와 무관한 작업은 독립 실행 보장 필요
- **"환경변수만 바꾸면 모드 전환 가능하다" 오해(과거):** tr_id가 하드코딩되어 있으면 코드 수정도 필요 → URL 기반 자동 분기 함수로 해결
- **"GitHub Secrets와 Cloudflare가 연동된다" 오해:** 완전 독립 시스템, 각각 별도로 환경변수 관리 필요
