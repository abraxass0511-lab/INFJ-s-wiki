# [[Flutter 포토 백업 앱 — v1~v11 개발 전체 기록]]

## 📌 한 줄 통찰 (The Karpathy Summary)
> Flutter 앱으로 안드로이드 갤러리 사진/동영상을 서버에 전송하고 백업 갤러리에서 보려면, photo_manager로 갤러리 접근 + Provider 상태관리 + SharedPreferences 영구저장 + video_player 재생 + 서버 HTTP Range 스트리밍(206 응답)이 필요하며, 가장 빈번한 버그는 서버-앱 간 DB 불일치(삭제된 항목 잔존)와 import 누락이다.

## 📖 구조화된 지식 (Synthesized Content)

### 추출된 패턴
- **photo_manager 전체 선택** — `AssetPathEntity.assetCountAsync`로 전체 개수 조회 후 페이지 단위 로드, 무한 스크롤 구현
- **Provider 패턴** — `TransferProvider`(전송 상태), `PhotoProvider`(백업 사진), `SyncProvider`(서버 동기화) 3개 Provider 분리
- **SharedPreferences 영구 저장** — 전송 완료된 asset ID를 `Set<String>`으로 관리, 앱 종료 후에도 초록 체크 유지
- **video_player + HTTP Range** — ExoPlayer는 Range 요청 필수, FastAPI에서 `StreamingResponse(status_code=206)` + `Content-Range` 헤더로 구현
- **전체 동기화 전략** — 델타 sync(lastSync 기반)는 삭제 반영 불가 → 매번 전체 데이터 가져와서 로컬 DB 교체
- **날짜별 그룹핑** — `CustomScrollView` + `SliverToBoxAdapter`(날짜 헤더) + `SliverGrid`(사진 그리드) 조합
- **CachedNetworkImage vs Image.network** — 서버 원본 파일 API(`/file/`)는 캐시 없이 `Image.network`가 더 안정적

### 대화 9: DB 불일치 + 전체 동기화 (v10)

**문제:** "전송한 사진들이 자동으로 백업 파일에 안들어가"

**근본 원인:** 
1. 서버 DB에서 삭제된 파일이 앱 로컬 DB에 남아있음
2. 델타 sync (`lastSync` 기반)는 삭제된 항목을 반영하지 못함
3. 업로드 시 `photos.buffer_path`가 저장되지 않아 `/file/` API에서 파일을 찾지 못함

**수정 3가지:**

1. **업로드 시 buffer_path 저장:**
```python
# upload.py — INSERT에 buffer_path 추가
buffer_file_path = buffer_path if storage_status == "buffer" else None
await db.execute("""INSERT INTO photos
   (..., buffer_path, ...) VALUES (..., ?, ...)""",
   (..., buffer_file_path, ...))
```

2. **전체 동기화:**
```dart
// sync_provider.dart
// 항상 전체 데이터 가져오기 (삭제 반영)
final data = await apiService.syncData(lastSyncAt: null);
```

3. **로컬 DB 전체 교체:**
```dart
// photo_provider.dart
await LocalDatabase.clearPhotos();  // 기존 데이터 삭제
await LocalDatabase.bulkUpsertPhotos(photoList);  // 서버 데이터로 교체
```

### 대화 10: 동영상 스트리밍 — ExoPlayer Range 요청 (v11)

**문제:** "아직도 동영상 재생이 안돼"
```
PlatformException(VideoError, Video player had error
androidx.media3.exoplayer.ExoPlaybackException: Source error, null, null)
```

**원인:** ExoPlayer(Android 비디오 플레이어)는 **HTTP Range 요청**을 보내지만, FastAPI의 `FileResponse`는 Range를 제대로 처리하지 못함

**수정 — StreamingResponse + 206 Partial Content:**
```python
from fastapi import Request
from starlette.responses import StreamingResponse

@router.get("/file/{photo_id}")
async def get_original_file(photo_id: str, request: Request = None):
    # ... 파일 경로 탐색 ...
    
    range_header = request.headers.get("range") if request else None
    
    if range_header and range_header.startswith("bytes="):
        # Range: bytes=0-1023 파싱
        parts = range_header.replace("bytes=", "").split("-")
        start = int(parts[0]) if parts[0] else 0
        end = int(parts[1]) if parts[1] else file_size - 1
        content_length = end - start + 1
        
        def iter_file():
            with open(file_path, "rb") as f:
                f.seek(start)
                remaining = content_length
                while remaining > 0:
                    data = f.read(min(8192, remaining))
                    if not data: break
                    remaining -= len(data)
                    yield data
        
        return StreamingResponse(
            iter_file(),
            status_code=206,
            media_type=media_type,
            headers={
                "Content-Range": f"bytes {start}-{end}/{file_size}",
                "Accept-Ranges": "bytes",
                "Content-Length": str(content_length),
            }
        )
```

**테스트 결과:**
```bash
curl -H "Range: bytes=0-1023" http://localhost:8000/api/photos/file/{id}
# HTTP 206, Content-Type: video/mp4, Size: 1024 ✅
```

## ⚠️ 모순 및 업데이트 (Contradictions & RL Update)

- **"FileResponse면 동영상도 된다" 오해:** ExoPlayer는 HTTP Range 요청(206 Partial Content) 필수 → `StreamingResponse`로 직접 구현해야 함
- **"델타 sync가 효율적이다" 오해:** 삭제된 항목을 반영하지 못함 → 전체 sync + 로컬 DB 교체가 더 안정적
- **"CachedNetworkImage가 항상 좋다" 오해:** 서버 원본 파일 API는 캐시 키 충돌 가능 → `Image.network`가 더 신뢰성 높음
- **"메모리 Set으로 상태 관리 충분하다" 오해:** 앱 종료 시 초기화됨 → SharedPreferences 필수
- **"DB INSERT에 모든 칼럼이 있다" 오해:** `buffer_path` 칼럼이 INSERT에 누락되어 `/file/` API 404 → 명시적으로 모든 경로 칼럼 저장
