# [[알파 투자 에이전트 — 승인 미실행 버그 수정 + 모의/실전 자동 분기]]

## 📌 한 줄 통찰 (The Karpathy Summary)
> Cloudflare Worker의 `scheduled()` 핸들러에서 단일 try-catch 구조는 A 섹션(체결 폴링)의 KIS API 토큰 만료 에러가 B 섹션(예약 매수/매도)까지 전파시켜 승인 건을 영구적으로 미실행 상태로 만들며, 이를 섹션별 독립 try-catch로 격리해야 하고, KIS 모의/실전 전환은 `KIS_BASE_URL`에 "vts" 포함 여부로 tr_id를 자동 분기하면 환경변수만 변경하여 코드 수정 없이 전환 가능하다.

## 📖 구조화된 지식 (Synthesized Content)

### 추출된 패턴
- **에러 격리 패턴** — 순차 실행되는 독립 작업들은 반드시 각각의 try-catch로 감싸야 함. 하나의 에러가 후속 작업을 차단하는 것을 방지
- **URL 기반 환경 분기** — 하드코딩된 ID를 환경변수(URL) 기반 조건 분기로 교체하면, 환경변수 변경만으로 모드 전환 가능
- **KIS API 토큰 특성** — 24시간 만료, 6시간 재발급 제한, `EGW00201` 에러로 토큰 만료 식별
- **Cloudflare Worker cron** — `*/5 13-21 * * 1-5` (UTC) = KST 22:00~06:00 매 5분, `scheduled()` 핸들러로 처리
- **KV 기반 예약 시스템** — `pending_approval`, `pending_sell`, `pending_fill_check` 3종 KV로 예약/추적 관리

### 대화 1: 승인 미실행 원인 분석

**사용자 요청:**
> 텔레그램에서 승인했는데, 매수/매도가 진행이 안되었어. 아침에 메시지가 와서 승인했는데 그 날 장이 개장되었을때 매수매도가 안되었어

**에이전트 분석:**
- `/debug` 엔드포인트로 KV 상태 조회 → `pending_approval` 잔존 확인
- KIS API 응답에서 `EGW00201` (토큰 만료) 에러 확인
- `scheduled()` 핸들러 코드 추적 → 단일 try-catch 구조 발견

**근본 원인:**
```javascript
// 수정 전: 단일 try-catch → A 에러 시 B 건너뜀
try {
  ① 체결 확인 (A)  ← KIS 토큰 에러 💥
  ② 승인 실행 (B)  ← 도달 못함 ❌
} catch {
  에러 로그만 남김
}
```

A 섹션(체결 확인)에서 KIS API 토큰 만료 에러(`EGW00201`)가 발생하면, B 섹션(예약 승인 실행)이 완전히 건너뛰어지고, 이것이 매 5분마다 반복되어 하루 종일 승인 건이 실행되지 않았음.

### 대화 2: 섹션별 독립 try-catch 수정

**수정 내용:**
```javascript
// 수정 후: 독립 try-catch → 각 섹션 독립 실행
try { A-0: stale cleanup } catch { console.log }   // 실패해도 OK
try { A: 체결 확인 }       catch { console.log }   // 실패해도 OK
try { B: 승인 실행 }       catch { sendMessage }   // 무조건 실행! ✅
```

**변경 파일:** `cloudflare-worker/worker.js`
- 기존 단일 try-catch → A-0, A, B 3개 독립 try-catch로 분리
- B 섹션의 catch에서는 `sendMessage()`로 에러 알림 발송 (사용자 인지 가능)
- A-0, A 섹션의 catch는 console.log만 (내부에서 자체 에러 처리 로직 존재)

### 대화 5: 모의/실전 자동 분기 구현

**사용자 요청:**
> 자동분기하면 왜 코드 수정이 0개야? 어떤 원리로 여기는 코드 15군데가 수정되는거야? → 원리 이해 후 → 수정해

**핵심 원리:**
```
모의투자 URL: https://openapi "vts" .koreainvestment.com:29443
실전투자 URL: https://openapi .koreainvestment.com:9443

→ URL에 "vts"가 있으면 모의, 없으면 실전
→ if문으로 tr_id 자동 선택
```

**worker.js 수정 — `getTrId()` 함수 추가:**
```javascript
function getTrId(env, type) {
  const isLive = !env.KIS_BASE_URL?.includes("vts");
  const map = {
    BALANCE: isLive ? "TTTS3012R" : "VTTS3012R",
    BUYING_POWER: isLive ? "TTTS3007R" : "VTTS3007R",
    FILLS: isLive ? "TTTS3035R" : "VTTS3035R",
    BUY: isLive ? "JTTT1002U" : "VTTT1002U",
    SELL: isLive ? "JTTT1006U" : "VTTT1006U",
  };
  return map[type] || type;
}
```

**alpha_trader.py 수정 — `_get_tr_id()` 메서드 추가:**
```python
def _get_tr_id(self, api_type):
    is_live = "vts" not in (self.base_url or "")
    tr_map = {
        "DEPOSIT": "CTRP6504R" if is_live else "VTRP6504R",
        "BUYING_POWER": "TTTS3007R" if is_live else "VTTS3007R",
        "BALANCE": "TTTS3012R" if is_live else "VTTS3012R",
        "BUY": "JTTT1002U" if is_live else "VTTT1002U",
        "SELL": "JTTT1006U" if is_live else "VTTT1006U",
        "FILLS": "TTTS3035R" if is_live else "VTTS3035R",
    }
    return tr_map.get(api_type, api_type)
```

**하드코딩 교체 (15곳):**
```javascript
// Before:
return await executeOrderWithRetry(env, "VTTT1002U", ...);

// After:
return await executeOrderWithRetry(env, getTrId(env, "BUY"), ...);
```

**side 판별도 개선:**

```javascript
// Before: 모의투자 ID 직접 비교
const side = trId === "VTTT1002U" ? "매수" : "매도";
```

## 🏗️ 시스템 아키텍처

```
텔레그램 → Cloudflare Worker (webhook)
              ├─ "승인" → handleApproval()
              │           ├─ 장중: executeApproval() → 즉시 매수/매도
              │           └─ 장외: KV에 pending_approval 저장
              ├─ "반려" → handleReject()
              ├─ "전량매도" → handleSellConfirm()
              ├─ "예약취소" → handleCancelReservation()
              └─ "시도" → GitHub Actions 트리거

Cloudflare Worker (cron */5분)
  scheduled() 핸들러:
    ├─ [A-0] Stale fill check 정리 (독립 try-catch)
    ├─ [A] 체결 확인 폴링 (독립 try-catch)
    └─ [B] 예약 주문 실행 (독립 try-catch)
        ├─ pending_sell → executeEmergencySell()
        └─ pending_approval → executeApproval()
```

### KV 상태 관리:
```
pending_approval  → 승인 예약 (장 닫힘 시 저장, 장 열리면 실행 후 삭제)
pending_sell      → 매도 예약 (장 닫힘 시 저장, 장 열리면 실행 후 삭제)
pending_fill_check → 체결 추적 (주문 접수 후 저장, 전종목 체결/장마감 시 삭제)
```

## ⚠️ 모순 및 업데이트 (Contradictions & RL Update)

- **"하나의 try-catch가 깔끔하다" 오해:** 순차적이지만 독립적인 작업들을 하나의 try-catch에 넣으면, 선행 작업의 에러가 후속 작업을 차단함 → 각 작업을 독립 try-catch로 격리해야 함
- **"토큰 에러는 일시적이다" 오해:** KIS 토큰은 24시간 만료, 6시간 재발급 제한이 있어서 특정 시간대에 반복 실패 가능 → 토큰 에러와 무관한 작업은 독립 실행 보장 필요
- **"환경변수만 바꾸면 모드 전환 가능하다" 오해(과거):** tr_id가 하드코딩되어 있으면 코드 수정도 필요 → URL 기반 자동 분기 함수로 해결
- **"GitHub Secrets와 Cloudflare가 연동된다" 오해:** 완전 독립 시스템, 각각 별도로 환경변수 관리 필요
