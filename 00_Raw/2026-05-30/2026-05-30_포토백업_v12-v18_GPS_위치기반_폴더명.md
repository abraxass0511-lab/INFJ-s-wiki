# [[포토 백업 앱 — v12~v18 GPS 위치 기반 폴더명 + 버퍼 관리]]

## 📌 한 줄 통찰 (The Karpathy Summary)
> 삼성 Galaxy 카메라는 EXIF에 GPS를 제대로 기록하지 않는 경우가 많아(NaN/0/0), 서버 측 EXIF 파싱만으로는 위치를 얻을 수 없다. 핵심 해결책은 앱에서 `AssetEntity.latlngAsync()`로 갤러리가 보정한 GPS를 가져와 서버에 전달하는 것이며, 이를 위해 Android `ACCESS_MEDIA_LOCATION` 권한이 반드시 필요하다.

## 📖 구조화된 지식 (Synthesized Content)

### 추출된 패턴
- **삼성 EXIF GPS 문제** — Galaxy S24+에서 EXIF GPS 값이 `NaN` 또는 `0/0(ZeroDivisionError)`로 기록됨. 삼성 갤러리 앱은 Wi-Fi/셀타워 기반으로 위치를 보정하지만, 이는 갤러리 내부 DB에만 저장되고 EXIF에는 반영 안 됨
- **ACCESS_MEDIA_LOCATION 필수** — Android 10+ 보안 정책으로, 이 권한 없이 `latlngAsync()` 호출 시 항상 null 반환. `READ_MEDIA_IMAGES`와 별도로 선언 필요
- **EXIF → 앱 GPS 폴백 전략** — 서버가 EXIF GPS를 먼저 확인하고, 없으면 앱이 Form 필드로 보낸 `latitude`/`longitude`를 사용
- **Nominatim 주소 필드 우선순위** — `amenity` → `building` → `road` 순서는 도로명이 잡힘. `leisure`(공원) → `suburb`(동네) → `city`(시) 순서가 더 의미 있음
- **버퍼 파일 미삭제 버그** — 외장하드 저장 시 `save_to_buffer` → `save_to_external` 후 버퍼 원본을 `unlink()` 안 하면 파일이 영구 잔존
- **Gemini API 무료 제한** — 일 500회, 분당 10회. 가족사진 용도로 충분하지만 100장 일괄 전송 시 속도 제한에 걸릴 수 있음. Nominatim은 일일 제한 없음(초당 1회만 준수)

### 대화 1: GPS 위치값이 "위치미상"으로 나오는 문제

**사용자 요청:**
> 위치값이 어떻게 나오는거야? 아직도 위치미상이네. 사진 찍으면 자동으로 위치값 나오는거 아니야?

**원인 분석:**
삼성 Galaxy S24+에서 EXIF GPS 데이터가 비정상:
```python
# EXIF에서 추출한 GPS 값
GPSLatitudeRef: '\x00'  # 비정상
GPSLatitude: (0/0, 0/0, 0/0)  # ZeroDivisionError
```

삼성 갤러리에서는 "대한민국 경기도 하남시"로 정상 표시 → 갤러리 내부 DB에서 보정된 값

**수정 (exif_service.py):**
```python
import math

# NaN GPS 좌표 필터링 (삼성 기기 대응)
if latitude is not None and (math.isnan(latitude) or math.isnan(longitude)):
    latitude = None
    longitude = None
    has_gps = False
```

### 대화 2: originFile vs file — EXIF 메타데이터 보존

**문제:** `asset.file`은 EXIF를 제거한 복사본을 반환

**수정 (gallery_service.dart):**
```dart
Future<File?> getFile(AssetEntity asset) async {
  // 원본 파일 우선 (EXIF GPS 보존)
  final originFile = await asset.originFile;
  if (originFile != null) return originFile;
  // 원본 파일 실패 시 일반 파일로 폴백
  return await asset.file;
}
```

### 대화 4: ACCESS_MEDIA_LOCATION 권한 누락 (v17→v18)

**문제:** v17에서 `latlngAsync()` 호출해도 항상 null 반환

**디버그 로그:**
```
🔍 GPS 디버그: EXIF has_gps=False, lat=None, lon=None
🔍 GPS 디버그: 앱 전송 lat=None, lon=None    ← 앱도 위치 못 가져옴
```

**근본 원인:** Android 10+ 보안 정책
```
READ_MEDIA_IMAGES  → 사진 파일 읽기     ✅ 있었음
ACCESS_MEDIA_LOCATION → 사진 GPS 읽기  ❌ 없었음!
```

**수정 (AndroidManifest.xml):**
```xml
<!-- 사진 위치 정보 접근 (Android 10+, GPS 좌표 읽기 필수) -->
<uses-permission android:name="android.permission.ACCESS_MEDIA_LOCATION" />
```

### 대화 5: 앱에서 GPS를 서버에 전달하는 파이프라인

**수정 — transfer_provider.dart (앱):**
```dart
// 앱에서 위치 데이터 가져오기 (삼성 갤러리 보정 포함)
double? lat;
double? lng;
try {
  final latLng = await item.asset.latlngAsync();
  if (latLng != null &&
      latLng.latitude != 0 && latLng.longitude != 0) {
    lat = latLng.latitude;
    lng = latLng.longitude;
  }
} catch (_) {}

final result = await apiService.uploadPhotoWithProgress(
  file.path,
  latitude: lat,
  longitude: lng,  // GPS를 Form 필드로 전달
);
```

**수정 — api_service.dart (앱):**
```dart
final map = <String, dynamic>{
  'file': await MultipartFile.fromFile(filePath),

if (latitude != null && longitude != null) {
  map['latitude'] = latitude.toString();

```

**수정 — upload.py (서버):**

async def upload_photo(
    file: UploadFile = File(...),

    # EXIF에 GPS 없으면 앱에서 보낸 위치로 보완
    if not has_gps and latitude is not None and longitude is not None:

### 대화 6: Nominatim 주소 필드 우선순위 개선

**문제:** 폴더명이 `미사강변남로`(도로명), `입주민 전용`(건물명) 등 부적절

**Nominatim 반환 데이터:**
```python
{
  'road': '미사강변남로',           # 도로명
  'suburb': '미사강변도시',         # 택지 이름
  'city_district': '미사1동',      # 행정동
  'city': '하남시',
  'province': '경기도',
  'country': '대한민국'
}
```

**기존 우선순위 (문제):**
`amenity` → `building` → `road` → `suburb` → `city`

**수정된 우선순위:**
```python
# 1순위: 공원, 관광지, 유명 장소
landmark = address.get("leisure") or address.get("tourism") or address.get("amenity")

# 2순위: 동네 이름
neighbourhood = address.get("neighbourhood") or address.get("quarter") or address.get("suburb")

# 3순위: 시/구
city = address.get("city") or address.get("county") or address.get("town")

# 조합 (도로명 단독 사용 방지)
if landmark: place_name = landmark
elif neighbourhood: place_name = neighbourhood
elif city: place_name = city
else: place_name = address.get("road") or location.address
```

# DB에 등록된 파일은 건너뜀 (upload API가 이관 담당)
orphan_files = [f for f in media_files if f.name not in db_files]

### 대화 8: 버퍼 파일 미삭제 버그

**문제:** 외장하드로 전달은 되는데 사진모음집에 파일이 남아있음

**원인:** `upload.py`에서 `save_to_buffer()` → `save_to_external()` 후 버퍼 원본을 삭제하지 않음

**수정 (upload.py):**
```python
temp_buffer = buffer_service.save_to_buffer(contents, file.filename)
saved_path = storage_service.save_to_external(temp_buffer, target_folder, file.filename)
if saved_path:
    storage_status = "external_drive"
    # 버퍼 파일 삭제 (외장하드에 저장 완료)
    try:
        buf_file = Path(temp_buffer)
        if buf_file.exists():
            buf_file.unlink()
    except Exception:
        pass
```

## ⚠️ 모순 및 업데이트 (Contradictions & RL Update)
- **"EXIF에 GPS가 있다" 오해:** 삼성 Galaxy S24+는 EXIF에 GPS를 NaN/0/0으로 기록하는 경우가 빈번 → 서버 EXIF 파싱만으로는 위치 확보 불가, 앱에서 `latlngAsync()` 폴백 필수
- **"READ_MEDIA_IMAGES면 사진 정보 전부 읽을 수 있다" 오해:** Android 10+에서 사진 GPS 접근은 `ACCESS_MEDIA_LOCATION` 별도 권한 필요 → 없으면 `latlngAsync()` 항상 null 반환
- **"originFile을 쓰면 EXIF가 보존된다" 오해:** originFile을 써도 삼성 기기는 EXIF GPS 자체가 비정상값 → EXIF 보존과 GPS 확보는 별개 문제
- **"Nominatim 첫 번째 주소 필드가 최선이다" 오해:** `amenity`/`building`/`road` 우선 시 도로명(`미사강변남로`)이나 건물명(`입주민 전용`)이 폴더명이 됨 → `leisure`(공원)/`suburb`(동네) 우선 필요
- **"Gemini API로 스마트 폴더명이 필요하다" 오해:** 무료 제한(일 500회, 분당 10회)과 복잡성 대비 Nominatim 동네명만으로도 충분 → 불필요한 의존성 제거
- **"외장하드 복사 후 버퍼는 자동 정리된다" 오해:** `save_to_external()`은 복사만 수행, 버퍼 원본 `unlink()` 누락 시 파일이 영구 잔존
- **"폴더 감시가 모든 파일을 이관하면 된다" 오해:** upload API가 처리한 파일까지 `직접전송` 폴더로 이중 복사됨 → DB 등록 여부 확인 후 이관 필요
