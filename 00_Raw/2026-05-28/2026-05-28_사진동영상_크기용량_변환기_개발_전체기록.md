# [[사진/동영상 크기 및 용량 변환기 — 브라우저 기반 개발 & GitHub Pages 배포]]

## 📌 한 줄 통찰 (The Karpathy Summary)
> Canvas API(이미지)와 FFmpeg.wasm(동영상)을 사용하면 서버 없이 브라우저만으로 파일 리사이즈/압축이 가능하며, GitHub Pages로 배포하면 노트북이 꺼져도 24시간 무료 호스팅이 된다. FFmpeg.wasm은 CORS/Worker 제약이 까다롭고, UMD 번들을 로컬에 자체 호스팅 + import.meta.url을 location.href로 교체하는 것이 가장 안정적인 해결책이다.

## 📖 구조화된 지식 (Synthesized Content)

### 추출된 패턴
- **클라이언트 사이드 처리** — 이미지/동영상 변환을 서버 없이 사용자 브라우저에서 수행 (개인정보 보호 + 비용 0원)
- **Canvas API 이미지 압축** — `toDataURL(format, quality)`의 quality를 이진 탐색하여 목표 용량 달성
- **FFmpeg.wasm Worker CORS** — CDN에서 로드한 FFmpeg의 내부 Worker가 same-origin 정책에 의해 차단됨
- **UMD 로컬 호스팅** — Worker 청크 파일(814.ffmpeg.js)을 로컬에 저장하면 CORS 문제 근본 해결
- **import.meta.url 파싱 에러** — UMD에 포함된 import.meta.url은 일반 script 태그에서 SyntaxError 발생 → location.href로 교체
- **GitHub Pages 무료 배포** — 정적 사이트를 `gh` CLI로 즉시 배포, 모바일/PC 어디서든 접속 가능

### 대화 1: 이미지 변환 웹사이트 요청

**사용자 요청:**
> 나는 사진의 픽셀과 용량을 변경하는 웹사이트를 만들거야.
> 예를 들어 1024×1024, 10kb 이미지를 넣으면 나의 요청사항인 32×32, 1kb 미만의 이미지를 적용해서 이미지를 새롭게 만들어서 다운받기 전에 제대로 되었는지 확인하고 내가 다운로드 클릭을 누르면 다운받을 수 있도록

**에이전트 작업:**
- `index.html` — 메인 UI 구조 생성 (업로드 → 설정 → 결과 3단계)
- `style.css` — 디자인 시스템 (CSS 변수 기반)
- `app.js` — Canvas API로 이미지 리사이즈 + quality 이진 탐색으로 목표 용량 달성
- 드래그 앤 드롭 업로드, 미리보기, 원본 vs 결과 비교, 다운로드 기능 구현

**핵심 기술 — Canvas API quality 이진 탐색:**
```javascript
let lo = 0.01, hi = 1.0;
while (hi - lo > 0.01) {
    let mid = (lo + hi) / 2;
    let dataUrl = canvas.toDataURL(format, mid);
    let size = dataURLtoBlob(dataUrl).size;
    if (size > maxSizeBytes) hi = mid;
    else lo = mid;
}
```

### 대화 10: 오류 #1 — CORS Worker 차단

**에러:** `Failed to construct 'Worker': Script at 'cdn.jsdelivr.net/.../worker.js' cannot be accessed from origin 'http://localhost:8080'`

**원인:** CDN에서 ESM import한 FFmpeg 내부 Worker가 cross-origin이라 차단

**수정:** classWorkerURL에 toBlobURL로 변환한 UMD worker 전달 → **실패 (undefined 에러)**

### 대화 11: 오류 #2 — undefined 에러

**수정:** ESM worker.js를 fetch → Blob → URL.createObjectURL → **실패 (초기화에서 멈춤)**

### 대화 12: 오류 #3 — "FFmpeg 초기화 중..."에서 멈춤

**사용자:** "되고 있는거야? 멈췄는데"

**원인:** ESM worker의 내부 import문이 blob URL 컨텍스트에서 resolve 불가

**근본 해결 — UMD 로컬 호스팅:**
1. `ffmpeg.js` — unpkg에서 UMD 번들 다운로드 → 로컬 저장
2. `814.ffmpeg.js` — Worker 청크 다운로드 → 같은 폴더에 저장
3. `<script src="ffmpeg.js">` 로컬 로드
4. video.js에서 글로벌 `FFmpegWASM` 사용

→ **실패 (FFmpegWASM is not defined)**

---

### 대화 13: 오류 #4 — FFmpegWASM is not defined

**원인:** ffmpeg.js에 `import.meta.url` 포함 → 일반 `<script>`에서 전체 파싱 실패(SyntaxError)

**최종 수정:** `import.meta.url` → `location.href` 교체

```javascript
// 변경 전 (SyntaxError)
new Worker(new URL(s, import.meta.url), {type:"module"})
// 변경 후 (정상)
new Worker(new URL(s, location.href), {type:"module"})
```

**재배포:**
```powershell
git add -A && git commit -m "fix: FFmpeg 로컬 UMD 호스팅으로 CORS 문제 해결"
git push origin master
```

## 🔧 FFmpeg.wasm 오류 해결 전체 기록 (4번 실패 → 성공)

| # | 시도 | 결과 | 실패 원인 |
|---|------|------|-----------|
| 1 | CDN ESM dynamic import | ❌ Worker CORS | Worker가 CDN cross-origin |
| 2 | toBlobURL → classWorkerURL | ❌ undefined | 파일 경로 불일치 |
| 3 | ESM worker fetch→Blob | ❌ 멈춤 | 내부 import문 resolve 불가 |
| 4 | UMD 로컬 (import.meta 포함) | ❌ 파싱 에러 | import.meta는 module 전용 |
| **5** | **UMD 로컬 + import.meta→location.href** | **✅ 성공** | **same-origin + 파싱 정상** |

### 최종 해결 체크리스트
```
□ ffmpeg.js (UMD 번들) 로컬 저장
□ 814.ffmpeg.js (Worker 청크) 같은 폴더에 저장
□ ffmpeg.js 내 import.meta.url → location.href 교체
□ <script src="ffmpeg.js"> (type="module" 아님!)
□ video.js에서 FFmpegWASM 글로벌 변수 사용
□ @ffmpeg/core(wasm)는 CDN에서 toBlobURL로 로드
□ @ffmpeg/util은 CDN UMD 스크립트 태그
```

## ⚠️ 모순 및 업데이트 (Contradictions & RL Update)
- **"FFmpeg.wasm은 CDN에서 바로 쓸 수 있다" 오해:** ESM dynamic import → Worker CORS 차단 → UMD 로컬 호스팅 필수
- **"classWorkerURL로 해결 가능" 부분적 오류:** ESM worker의 내부 import가 blob URL에서 작동 안 함
- **"import.meta.url은 UMD에 없다" 실수:** 원본 UMD 빌드에 포함되어 있어 일반 script에서 SyntaxError
- **"toBlobURL이 만능" 오해:** Worker 내부의 추가 import까지는 해결 못함
- **coi-serviceworker vs CORS 혼동:** coi-serviceworker = COEP/COOP(SharedArrayBuffer), CORS Worker = same-origin 정책 — 별개 문제
- **4번 실패 교훈:** "로컬 파일 호스팅 + UMD 빌드 + import.meta 제거" 3가지 조건 모두 충족 필요
