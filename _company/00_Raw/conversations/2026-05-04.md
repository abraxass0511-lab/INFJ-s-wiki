| :--- | :--- | :--- | :--- |
| **실행 가능성 극대화** | 경험을 기록만 하고 실제 행동으로 연결하지 못함. (지식 사장) | **'How-to' 중심의 구조화:** 단순 사실 나열이 아닌, 특정 결과를 도출하기 위한 단계별 프로세스 및 의사결정 프레임워크 제공. | 실행 계획 템플릿, 의사결정 트리(Decision Tree), 액션 플랜 템플릿 |

| **자동화 및 효율** | 문서화 과정 자체가 또 다른 노동이 되어 지속성이 떨어짐. (노동 집약적) | **템플릿 기반 자동화:** 사용자가 최소한의 입력만으로 고품질의 구조를 즉시 생성할 수 있는 '템플릿' 제공. | 지식 템플릿 패키지, 자동화된 문서 생성 워크플로우 |

# 🚀 MVP 템플릿 A: 일상 경험을 실행 가능한 자산으로 전환하는 구조화 방법론 초안

본 문서는 '일상 경험에서 실행 가능성을 높이는 구조화된 방법론'의 핵심 프레임워크를 제시하며, 사용자의 경험(데이터)을 단순한 기록이 아닌 **실험과 성장의 사이클**로 전환하는 것을 목표로 합니다.

## 🎯 핵심 프레임워크: 경험 기반 자산 전환 사이클 (The Experience-to-Asset Cycle)

모든 경험은 다음의 4단계 순환 고리를 통해 실행 가능한 지식으로 변환됩니다.

### Phase 1: 관찰 및 기록 (Observation & Capture)
**목표:** 감정적 기록을 객관적인 데이터로 전환한다.
*   **핵심 활동:** 일상에서 발생하는 경험(행동, 결과, 감정)을 있는 그대로 포착하고 구조화한다.
*   **산출물:** 원시 데이터 로그 (Raw Log)

### Phase 2: 가설 설정 및 설계 (Hypothesis & Design)
**목표:** 기록된 경험에 의미를 부여하고 다음 행동의 방향을 설정한다.
*   **핵심 활동:** 관찰된 현상에 대해 '왜?'라는 질문을 던지고, 실행 가능한 가설(Hypothesis)을 설정한다. 이 가설을 검증하기 위한 구체적인 실험 설계(Experiment Design)를 세운다.
*   **산출물:** 실험 계획서 (Experiment Plan), 핵심 가설 정의

### Phase 3: 실행 및 측정 (Execution & Measurement)
**목표:** 가설을 실제 행동으로 테스트하고 객관적인 결과를 확보한다.
*   **핵심 활동:** 설계한 실험대로 행동을 실행하고, 그 과정과 결과를 정량적/정성적으로 기록한다. (A/B 테스트 또는 반복 학습 사이클 적용)
*   **산출물:** 실행 로그 (Execution Log), 측정 데이터 (Metrics)

### Phase 4: 구조화 및 일반화 (Structuring & Generalization)
**목표:** 측정된 결과를 지식으로 승화시키고, 다른 경험에 적용할 수 있는 방법론을 도출한다.
*   **핵심 활동:** 실행 로그를 분석하여 핵심 인사이트(Insight)를 추출하고, 이 인사이트를 일반적인 규칙이나 프레임워크로 추상화한다. (경험들을 연결하는 지식 맵 구축)
*   **산출물:** 구조화된 방법론 (Structured Framework), 실행 가능한 자산 (Actionable Asset)

## 📚 MVP 템플릿 A: 세부 목차 및 구성 요소

이 프레임워크를 실제 적용하기 위한 구체적인 템플릿의 목차는 다음과 같습니다.

### I. 경험 기록 모듈 (Phase 1: 관찰 및 기록)
*   **A. 원시 로그 입력:** 날짜, 상황, 감정 상태, 발생한 행동 (What happened?)
*   **B. 맥락 부여:** 해당 경험이 발생한 환경(Context), 참여자(Actor), 사용된 자원(Resources) 명시

### II. 실험 설계 모듈 (Phase 2: 가설 설정 및 설계)
*   **A. 핵심 질문 정의:** 이 경험에서 내가 알고 싶은 것은 무엇인가? (Pain Point/Goal 정의)
*   **B. 실행 가설 (Hypothesis):** "만약 [X]를 한다면, [Y]라는 결과가 나올 것이다."
*   **C. 실험 설계:** 독립 변수(내가 조작할 것), 종속 변수(측정할 것), 통제 변수(고정할 것) 설정.

### III. 실행 및 측정 모듈 (Phase 3: 실행 및 측정)
*   **A. 실행 단계 기록:** 계획대로 행동을 수행한 과정 상세 기록.
*   **B. 결과 데이터 입력:** 정량적 지표 (숫자, 시간, 비용 등), 정성적 피드백 (느낀 점).
*   **C. 실패/성공 분석:** 가설이 맞았는지 틀렸는지에 대한 객관적인 평가.

### IV. 구조화 및 일반화 모듈 (Phase 4: 구조화 및 일반화)
*   **A. 핵심 인사이트 추출:** 측정된 데이터에서 도출된 가장 중요한 교훈(Lesson Learned) 3가지 정의.
*   **B. 방법론 프레임워크 구축:** 추출된 인사이트를 다른 경험에도 적용할 수 있는 보편적인 규칙(Rule) 또는 단계별 프로세스(Process)로 변환하여 문서화. (예: 의사결정 트리, 반복 학습 사이클 등)
*   **C. 지식 연결 맵:** 이 경험이 과거의 어떤 지식과 연결되는지 시각적으로 매핑하는 구조.

## 경험 기반 자산 전환 사이클 자동화 MVP 시스템 설계

Writer가 제시한 '경험 기반 자산 전환 사이클' 프레임워크를 자동화하기 위한 최소 기능 제품(MVP) 시스템 설계 및 초기 구조를 정의합니다. 이 시스템은 **일상 경험을 입력받아, 구조화된 지식 자산으로 변환하는 파이프라인**을 목표로 합니다.

### 1. 기술 스택 제안 (Technology Stack Proposal)

| 영역 | 추천 기술 | 선정 이유 |
| :--- | :--- | :--- |
| **백엔드/파이프라인** | Python (FastAPI/Pandas) | 데이터 처리, API 구축에 강력하며, 기존 LLM 및 데이터 라이브러리와의 통합이 용이함. |
| **데이터베이스** | PostgreSQL | 구조화된 지식(템플릿, 메타데이터)과 비정형 경험 데이터를 안정적으로 저장하고 관계를 관리하기 위함. |
| **LLM/AI 엔진** | OpenAI API (GPT-4o) 또는 자체 Fine-tuned 모델 | 경험을 분석하고, 추상적인 개념을 구체적인 실행 가능한 프레임워크로 변환하는 핵심 추론 능력 제공. |

### 2. 핵심 기능별 API 및 데이터 파이프라인 초기 구조 정의

시스템은 크게 **입력(Ingestion) $\rightarrow$ 처리(Processing) $\rightarrow$ 출력(Output)**의 세 단계로 구성됩니다.

#### A. 데이터 파이프라인 (Data Pipeline Flow)

1.  **Input Layer (경험 수집):** 사용자의 비정형 경험 데이터 입력 (텍스트, 음성 녹취록, 메모 등).
2.  **Processing Layer (지식 구조화):** LLM을 사용하여 입력된 경험에서 핵심 원칙(Principle), 실행 단계(Steps), 결과(Outcome)를 추출하고, 이를 사전에 정의된 '자산 전환 프레임워크'에 매핑합니다.
3.  **Storage Layer (자산 저장):** 구조화된 지식 자산(템플릿, 방법론)과 원본 경험을 분리하여 저장합니다.
4.  **Output Layer (자산 생성):** 추출된 구조를 바탕으로 판매 가능한 형태의 최종 결과물(템플릿, 가이드)을 생성합니다.

#### B. 핵심 API 엔드포인트 정의 (Initial API Endpoints)

| Endpoint | Method | 기능 설명 | 입력 데이터 (Request Body) | 출력 데이터 (Response Body) |
| :--- | :--- | :--- | :--- | :--- |
| `/api/v1/ingest` | POST | 새로운 경험 데이터를 시스템에 등록하고 초기 분석을 요청 | `experience_text`, `source_context` | `job_id`, `status`, `initial_analysis` |
| `/api/v1/process/{job_id}` | GET | 특정 작업의 처리 상태 및 상세 결과 조회 | (없음) | `status`, `structured_data`, `extracted_principles`, `suggested_asset_type` |
| `/api/v1/template/generate` | POST | 구조화된 데이터를 기반으로 판매용 템플릿 생성 요청 | `framework_id`, `extracted_principles`, `desired_format` | `asset_draft` (JSON/Markdown 형식) |
| `/api/v1/asset/{asset_id}` | GET | 최종 생성된 지식 자산(템플릿) 조회 | (없음) | `asset_content`, `metadata`, `version` |

### 3. 초기 데이터 모델 구조 (Database Schema Proposal)

PostgreSQL에 저장될 핵심 테이블의 초기 스키마를 제안합니다.

**Table: `experiences` (경험 원본 저장)**
*   `experience_id` (PK)
*   `user_id` (FK, 사용자 식별자)
*   `raw_data` (JSONB 또는 TEXT, 원본 입력 데이터)
*   `ingestion_timestamp` (타임스탬프)

**Table: `knowledge_assets` (구조화된 자산 저장)**
*   `asset_id` (PK)
*   `framework_id` (FK, 적용된 방법론 ID)
*   `title` (String, 생성된 자산의 제목)
*   `content` (JSONB/TEXT, 최종 구조화된 템플릿 내용)
*   `metadata` (JSONB, 핵심 원칙, 실행 단계, 예상 결과 등 메타데이터)
*   `status` (Enum: Draft, Review, Published)

**Table: `pipeline_jobs` (파이프라인 작업 추적)**
*   `job_id` (PK)
*   `experience_id` (FK, 연결된 경험 ID)
*   `status` (Enum: Pending, Processing, Failed, Completed)
*   `llm_prompt_used` (TEXT, 사용된 핵심 프롬프트 버전)
*   `processing_log` (JSONB, LLM 호출 기록 및 에러 로그)
*   `created_at`, `updated_at`
